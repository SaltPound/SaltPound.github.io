## IoC Frameworks: Overview and Comparison

![IoC Containers]({{ site.url }}{{ site.baseurl }}/assets/images/di-aop/ioc-containers.png){: .align-center}

The IoC container landscape in .NET has changed significantly over the past
decade. Ninject, Unity, and Spring.NET have ceded ground to lighter, faster
solutions. Yet the core concepts — registration, resolution, interception —
remain unchanged.

### Classic frameworks (historical perspective)

| IoC framework | Strengths | Weaknesses |
|---|---|---|
| **Ninject** | Interception, lightweight (~122 KB), extensible (MVC, WCF) | Slow resolution |
| **Castle Windsor** | Mature, understands Decorator, typed factories, commercial support | Quirky API in places |
| **Unity** | Interception, good docs, consistent API | Weak lifetime management, no convention-based API |
| **Spring.NET** | Interception, extensive docs, commercial support | XML-centric, no convention-based API, limited auto-wiring |
| **Autofac** | Easy-to-learn API, commercial support | No built-in interception, partial support for custom lifetimes |

### Current frameworks (2026)

| IoC framework | Status | When to use |
|---|---|---|
| **Microsoft.Extensions.DI** | ✅ Built into .NET, actively developed | Most new projects |
| **Autofac** | ✅ Active, rich API | Complex scenarios with modules |
| **Castle Windsor** | ✅ Active | Legacy projects, enterprise with interception |
| **Scrutor** | ✅ Active | Convention-based registration on top of MS DI |
| **DryIoc** | ✅ Fastest in benchmarks | High-load, performance-critical |
| **Ninject** | ⚠️ Slow development | Legacy only |
| **Unity** | ❌ Abandoned by Microsoft | Migrate to MS DI |
| **Spring.NET** | ❌ Not maintained | Migrate |

## IoC Container Performance

![IoC Performance]({{ site.url }}{{ site.baseurl }}/assets/images/di-aop/ioc-performance.png){: .align-center}

The performance question around IoC containers is often overblown.

> *"In 99% of use cases you will never push your IoC container close to the
> limits the benchmarks reveal."*

Still, it helps to know the order of magnitude. Current benchmarks (2024) show:

| Container | Singleton (ops/sec) | Transient (ops/sec) |
|---|---|---|
| **DryIoc** | ~200M | ~80M |
| **MS DI** | ~150M | ~60M |
| **Autofac** | ~50M | ~20M |
| **Castle Windsor** | ~30M | ~15M |
| **Ninject** | ~5M | ~2M |

Practical takeaway: if your bottleneck is dependency resolution — and not the
database, the network, or business logic — you have a very unusual
application. Choose a container for API ergonomics and ecosystem fit, not for
benchmark digits.

## Composition Root

![Composition Root]({{ site.url }}{{ site.baseurl }}/assets/images/di-aop/composition-root-1.png){: .align-center}

The **Composition Root** is the single place in your application where the
entire dependency graph is assembled. It's an architectural pattern, not just
"where configuration lives".

An analogy: a plant's root system is both the entry point for nutrients and
the place the whole structure grows from. The IoC container is the root of
your application — everything else grows out from it.

### Rules for a good Composition Root

1. **One per application.** Not several places, not "another container for
   this module".
2. **As close to the entry point as possible.** `Program.cs` in .NET 6+,
   `Startup.cs` in earlier versions.
3. **No Service Locator.** The container must not leak into business logic.
4. **Prefer explicit registration** over convention-based for critical
   components.

![Composition Root (2)]({{ site.url }}{{ site.baseurl }}/assets/images/di-aop/composition-root-2.png){: .align-center}

> *"Explicit is always better than implicit."*

### Composition Root in .NET 8+ (minimal API)

```csharp
// Program.cs IS the Composition Root
var builder = WebApplication.CreateBuilder(args);

// --- Infrastructure ---
builder.Services.AddDbContext<AppDbContext>(opt =>
    opt.UseNpgsql(builder.Configuration.GetConnectionString("Default")));

builder.Services.AddStackExchangeRedisCache(opt =>
    opt.Configuration = builder.Configuration["Redis:Connection"]);

// --- Repositories ---
builder.Services.AddScoped<IOrderRepository, EfOrderRepository>();
builder.Services.AddScoped<IProductRepository, EfProductRepository>();

// Decorators via Scrutor (caching, transparently)
builder.Services.Decorate<IProductRepository, CachedProductRepository>();

// --- Business logic ---
builder.Services.AddScoped<IOrderService, OrderService>();
builder.Services.AddScoped<IInventoryService, InventoryService>();
builder.Services.AddMediatR(cfg => cfg.RegisterServicesFromAssembly(typeof(Program).Assembly));

// --- Pipeline behaviors (AOP via MediatR) ---
builder.Services.AddTransient(typeof(IPipelineBehavior<,>), typeof(ValidationBehavior<,>));
builder.Services.AddTransient(typeof(IPipelineBehavior<,>), typeof(LoggingBehavior<,>));
builder.Services.AddTransient(typeof(IPipelineBehavior<,>), typeof(CachingBehavior<,>));

// --- External services ---
builder.Services.AddHttpClient<IShippingClient, FedExShippingClient>(client =>
    client.BaseAddress = new Uri(builder.Configuration["FedEx:BaseUrl"]!));

// --- Modern dependencies ---
builder.Services.AddScoped<IAIAssistant, OpenAIAssistant>();
builder.Services.AddFeatureManagement();

var app = builder.Build();
```

### Composition Root with Autofac (modular registration)

Autofac introduces the concept of **modules** — named groups of registrations.
Handy for larger applications:

```csharp
// Infrastructure module
public class InfrastructureModule : Module
{
    protected override void Load(ContainerBuilder builder)
    {
        builder.RegisterType<EfOrderRepository>()
            .As<IOrderRepository>()
            .InstancePerLifetimeScope();

        builder.RegisterType<EfProductRepository>()
            .As<IProductRepository>()
            .InstancePerLifetimeScope()
            .EnableInterfaceInterceptors()          // interception!
            .InterceptedBy(typeof(CacheInterceptor));

        builder.RegisterType<CacheInterceptor>().AsSelf();
    }
}

// Application module
public class ApplicationModule : Module
{
    protected override void Load(ContainerBuilder builder)
    {
        builder.RegisterAssemblyTypes(ThisAssembly)
            .Where(t => t.Name.EndsWith("Service"))
            .AsImplementedInterfaces()
            .InstancePerLifetimeScope();
    }
}

// Program.cs
builder.Host.UseServiceProviderFactory(new AutofacServiceProviderFactory());
builder.Host.ConfigureContainer<ContainerBuilder>(container =>
{
    container.RegisterModule<InfrastructureModule>();
    container.RegisterModule<ApplicationModule>();
});
```

## The Register → Resolve → Release Pattern

The RRR pattern describes the mandatory order of interaction with an IoC
container:

**Register** — set up the container: which classes it knows about, how
abstractions map to concrete types, how components relate.

**Resolve** — resolve the root component. One object graph is built per
request for one type.

**Release** — release components. Every object graph resolved in the previous
phase must be released when it's no longer needed. This matters especially
for `IDisposable` components.

```csharp
// Register (once, at startup)
var builder = WebApplication.CreateBuilder(args);
builder.Services.AddScoped<IOrderService, OrderService>();
builder.Services.AddScoped<IUnitOfWork, EfUnitOfWork>();
var app = builder.Build();

// Resolve (per HTTP request — the framework handles it automatically)
app.MapPost("/orders", async (CreateOrderCommand cmd, IOrderService svc) =>
    await svc.CreateAsync(cmd));

// Release — ASP.NET Core does this automatically at the end of the scope (request)
// All IDisposable Scoped and Transient components are released
app.Run();
```

### Antipattern: Service Locator

```csharp
// ❌ Service Locator — antipattern
public class OrderService
{
    public async Task ProcessAsync(int orderId)
    {
        // Reaching into the container from business logic
        var repo = ServiceLocator.Current.GetInstance<IOrderRepository>();
        var order = await repo.GetByIdAsync(orderId);
    }
}

// ✅ Correct — dependencies passed in from outside
public class OrderService
{
    private readonly IOrderRepository _repo;

    public OrderService(IOrderRepository repo) => _repo = repo;

    public async Task ProcessAsync(int orderId)
    {
        var order = await _repo.GetByIdAsync(orderId);
    }
}
```

**Why is Service Locator an antipattern?**

- Dependencies are hidden — you can't tell from a class's constructor what it
  actually needs.
- Testing gets harder — you have to configure a global locator.
- It violates the principle of explicit dependencies.

## Interception

![Interception]({{ site.url }}{{ site.baseurl }}/assets/images/di-aop/interception.png){: .align-center}

**Interception** is a mechanism for adding behavior (an aspect) to a method
without changing the method's code. It's implemented via a **Dynamic Proxy** —
a wrapper class generated at runtime that implements the same interface as
the target object.

How it works:

1. Calling code asks the container for `IMyInterface`.
2. The container returns `ProxyInterceptor : IMyInterface` instead of
   `MyClass : IMyInterface`.
3. The proxy intercepts the call, runs the aspect (logging, caching,
   authorization).
4. The proxy then calls the real method and returns the result.

### Castle Windsor: Dynamic Proxy Interception

```csharp
// Define an interceptor
public class LoggingInterceptor : IInterceptor
{
    private readonly ILogger _logger;

    public LoggingInterceptor(ILogger logger) => _logger = logger;

    public void Intercept(IInvocation invocation)
    {
        _logger.LogInformation("Calling: {Method}({Args})",
            invocation.Method.Name,
            string.Join(", ", invocation.Arguments));

        var sw = Stopwatch.StartNew();
        try
        {
            invocation.Proceed();   // invoke the real method
            _logger.LogInformation("Completed: {Method} [{Elapsed}ms]",
                invocation.Method.Name, sw.ElapsedMilliseconds);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Error in {Method}", invocation.Method.Name);
            throw;
        }
    }
}

// Registration with interception
var container = new WindsorContainer();
container.Register(
    Component.For<LoggingInterceptor>(),
    Component.For<IOrderService>()
        .ImplementedBy<OrderService>()
        .Interceptors<LoggingInterceptor>()
);

// Usage — transparent, no changes at the call site
var service = container.Resolve<IOrderService>();
service.CreateOrder(cmd);  // ← automatically logged
```

### Autofac: interception via Castle DynamicProxy

```csharp
// In Autofac, interception goes through Castle DynamicProxy integration
builder.RegisterType<OrderService>()
    .As<IOrderService>()
    .EnableInterfaceInterceptors()
    .InterceptedBy(typeof(LoggingInterceptor), typeof(CacheInterceptor));

builder.Register(c => new LoggingInterceptor(c.Resolve<ILogger<LoggingInterceptor>>()));
builder.Register(c => new CacheInterceptor(c.Resolve<IMemoryCache>()));
```

### Microsoft.Extensions.DI + DispatchProxy (built-in)

For those who want interception without pulling in extra libraries —
`DispatchProxy` from .NET Standard:

```csharp
public class LoggingProxy<T> : DispatchProxy where T : class
{
    private T _decorated = null!;
    private ILogger _logger = null!;

    protected override object? Invoke(MethodInfo? targetMethod, object?[]? args)
    {
        _logger.LogInformation("→ {Method}", targetMethod?.Name);
        var result = targetMethod?.Invoke(_decorated, args);
        _logger.LogInformation("← {Method}", targetMethod?.Name);
        return result;
    }

    public static T Create(T decorated, ILogger logger)
    {
        var proxy = Create<T, LoggingProxy<T>>() as LoggingProxy<T>;
        proxy!._decorated = decorated;
        proxy!._logger = logger;
        return (proxy as T)!;
    }
}

// Usage
services.AddScoped<IOrderService>(sp =>
    LoggingProxy<IOrderService>.Create(
        sp.GetRequiredService<OrderService>(),
        sp.GetRequiredService<ILogger<OrderService>>()
    ));
```

## IL Code Weaving

![IL Weaving]({{ site.url }}{{ site.baseurl }}/assets/images/di-aop/il-weaving.png){: .align-center}

**IL Weaving** is a fundamentally different approach from Dynamic Proxy.
Aspects are added not at runtime but at build time (a post-compile step):

1. The compiler produces an ordinary assembly (`.dll` / `.exe`).
2. An AOP post-processor (PostSharp, Metalama) reads the assembly and finds
   annotated methods.
3. The post-processor weaves aspect code directly into the IL.
4. The output is a finished assembly with the aspects already woven in.

**Advantages over Dynamic Proxy:**

- No runtime overhead from creating proxy objects.
- Works with non-interface classes and even static methods.
- Intercepts both `virtual` and non-`virtual` methods.
- Compile-time verification of aspect correctness.

**Drawbacks:**

- Complicates the build pipeline.
- Magic in the code: what you see isn't what you wrote.
- Commercial license for serious use (PostSharp).

## PostSharp and Metalama

![PostSharp and Metalama]({{ site.url }}{{ site.baseurl }}/assets/images/di-aop/postsharp-metalama.png){: .align-center}

**PostSharp** is the most mature IL weaver for .NET, around since 2004. In
2022 the same authors released **Metalama** — a modern successor built on
Roslyn Source Generators instead of IL Weaving.

### PostSharp: OnMethodBoundaryAspect

```csharp
// Define the aspect
[Serializable]
public class LoggingAspect : OnMethodBoundaryAspect
{
    public override void OnEntry(MethodExecutionArgs args)
    {
        Console.WriteLine($"→ Entering: {args.Method.Name}");
        args.MethodExecutionTag = Stopwatch.StartNew();
    }

    public override void OnExit(MethodExecutionArgs args)
    {
        var sw = (Stopwatch)args.MethodExecutionTag;
        Console.WriteLine($"← Leaving: {args.Method.Name} [{sw.ElapsedMilliseconds}ms]");
    }

    public override void OnException(MethodExecutionArgs args)
    {
        Console.WriteLine($"✗ Error: {args.Method.Name} — {args.Exception.Message}");
        args.FlowBehavior = FlowBehavior.Continue;  // suppress the exception
    }
}

// Apply to a method
public class OrderService
{
    [LoggingAspect]
    public Order CreateOrder(CreateOrderCommand cmd)
    {
        // pure business code, not a single line of logging
        return new Order(cmd.CustomerId, cmd.Items);
    }
}
```

```csharp
// OnExceptionAspect — a specialised aspect for exceptions
[Serializable]
public class RetryAspect : OnExceptionAspect
{
    public int Attempts { get; set; } = 3;

    public override void OnException(MethodExecutionArgs args)
    {
        if (args.Exception is TransientException && --Attempts > 0)
        {
            Thread.Sleep(200);
            args.FlowBehavior = FlowBehavior.Retry;
        }
    }
}

public class ExternalApiClient
{
    [RetryAspect(Attempts = 5)]
    public async Task<ApiResponse> CallAsync(string endpoint)
    {
        // the network may fail — the aspect handles retries
        return await _httpClient.GetFromJsonAsync<ApiResponse>(endpoint);
    }
}
```

### Multicast: apply an aspect across the whole assembly

```csharp
// AssemblyInfo.cs — applies the aspect to every public method in the assembly
[assembly: LoggingAspect(
    AttributeTargetTypes = "MyApp.Services.*",
    AttributeTargetMemberAttributes = MulticastAttributes.Public)]

[assembly: RetryAspect(
    AttributeTargetTypes = "MyApp.Infrastructure.Clients.*",
    Attempts = 3)]
```

### Metalama (2026): Roslyn-based, modern replacement

Metalama operates at the Roslyn level rather than IL, which means better IDE
integration and compile-time diagnostics:

```csharp
// A logging aspect in Metalama
public class LogAttribute : OverrideMethodAspect
{
    public override dynamic? OverrideMethod()
    {
        var methodName = meta.Target.Method.Name;
        Console.WriteLine($"→ {methodName}({string.Join(", ", meta.Target.Parameters.Values)})");

        try
        {
            var result = meta.Proceed();
            Console.WriteLine($"← {methodName} = {result}");
            return result;
        }
        catch (Exception ex)
        {
            Console.WriteLine($"✗ {methodName}: {ex.Message}");
            throw;
        }
    }
}

// Applied just like any attribute
public class OrderService
{
    [Log]
    public Order CreateOrder(CreateOrderCommand cmd) => new Order(cmd);
}
```

**Metalama vs PostSharp:**

| | PostSharp | Metalama |
|---|---|---|
| Mechanism | IL weaving (post-compile) | Roslyn Source Generators (compile-time) |
| IDE support | Good | Excellent (Roslyn-native) |
| Diagnostics | Runtime | Compile-time |
| License | Paid | Free tier + paid |
| .NET compatibility | .NET Framework + .NET | .NET 6+ |

## AOP Frameworks: the full map (2026)

### Via interception (runtime proxy)

```
┌─────────────────────────────────────────────────────┐
│          RUNTIME INTERCEPTION (Dynamic Proxy)        │
├────────────────┬────────────────┬───────────────────┤
│ Castle Windsor │    Autofac     │  DispatchProxy    │
│  (built-in)    │  (+Castle DP)  │ (built into .NET) │
└────────────────┴────────────────┴───────────────────┘
  Pros: easy to set up, works through interfaces
  Cons: virtual/interface methods only, small overhead
```

### Via IL weaving (compile-time)

```
┌─────────────────────────────────────────────────────┐
│              IL WEAVING (Build-time)                 │
├────────────────────────┬────────────────────────────┤
│       PostSharp        │         Metalama            │
│      (classic)         │    (Roslyn, modern)         │
└────────────────────────┴────────────────────────────┘
  Pros: any methods, zero runtime overhead, faster
  Cons: magic in the code, harder to debug, paid
```

### Via pipeline (modern .NET)

```
┌─────────────────────────────────────────────────────┐
│            PIPELINE / DECORATOR PATTERN              │
├──────────────┬──────────────┬───────────────────────┤
│ ASP.NET Core │   MediatR    │  Scrutor Decorators   │
│  Middleware  │  Behaviors   │                       │
└──────────────┴──────────────┴───────────────────────┘
  Pros: explicit, testable, no dependencies
  Cons: write it by hand, more code
```

## When to pick what

| Scenario | Recommendation |
|---|---|
| New ASP.NET Core project | MS DI + Scrutor + MediatR Behaviors |
| Need interception without PostSharp | Castle Windsor, or Autofac + Castle DynamicProxy |
| Audit & compliance in a financial system | Metalama (compile-time guarantees) |
| Legacy project on .NET Framework | PostSharp or Castle Windsor |
| CQRS with commands and queries | MediatR Pipeline Behaviors |
| Simple caching | Decorator + Scrutor |
| Very high load | DryIoc + minimal aspects |

## Worked example: the full stack

Pulling it together: MS DI + Autofac + MediatR + Decorator + Castle
Interceptor for a typical enterprise service.

```csharp
// 1. Interface and implementation
public interface IProductService
{
    Task<ProductDto> GetProductAsync(int id);
    Task<IReadOnlyList<ProductDto>> SearchAsync(string query);
}

public class ProductService : IProductService
{
    private readonly IProductRepository _repo;
    private readonly IAIAssistant _ai;

    public ProductService(IProductRepository repo, IAIAssistant ai)
    {
        _repo = repo;
        _ai = ai;
    }

    public async Task<ProductDto> GetProductAsync(int id)
    {
        var product = await _repo.GetByIdAsync(id)
            ?? throw new NotFoundException($"Product {id} not found");
        return product.ToDto();
    }

    public async Task<IReadOnlyList<ProductDto>> SearchAsync(string query)
    {
        // AI-enhanced search
        var enrichedQuery = await _ai.CompleteAsync(
            $"Expand search query with synonyms: {query}");
        return (await _repo.SearchAsync(enrichedQuery))
            .Select(p => p.ToDto())
            .ToList();
    }
}

// 2. Caching decorator (AOP via Scrutor)
public class CachedProductService : IProductService
{
    private readonly IProductService _inner;
    private readonly IMemoryCache _cache;
    private static readonly TimeSpan _ttl = TimeSpan.FromMinutes(5);

    public CachedProductService(IProductService inner, IMemoryCache cache)
    {
        _inner = inner;
        _cache = cache;
    }

    public Task<ProductDto> GetProductAsync(int id)
        => _cache.GetOrCreateAsync($"product:{id}",
            entry => { entry.AbsoluteExpirationRelativeToNow = _ttl;
                       return _inner.GetProductAsync(id); });

    public Task<IReadOnlyList<ProductDto>> SearchAsync(string query)
        => _inner.SearchAsync(query); // search not cached
}

// 3. Composition Root
builder.Services.AddMemoryCache();
builder.Services.AddScoped<IProductService, ProductService>();
builder.Services.Decorate<IProductService, CachedProductService>();

// 4. MediatR Behavior for validation (AOP for commands)
public class GetProductQuery : IRequest<ProductDto>
{
    public int ProductId { get; init; }
}

public class GetProductQueryHandler : IRequestHandler<GetProductQuery, ProductDto>
{
    private readonly IProductService _service;
    public GetProductQueryHandler(IProductService service) => _service = service;

    public Task<ProductDto> Handle(GetProductQuery request, CancellationToken ct)
        => _service.GetProductAsync(request.ProductId);
}

// 5. Endpoint — receives the already-cached, already-logged result
app.MapGet("/products/{id}", async (int id, IMediator mediator) =>
    await mediator.Send(new GetProductQuery { ProductId = id }));
```
